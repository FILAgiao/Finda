# ADR-001: 移动端采用原生开发

## 状态
- **提案人**: @hufei
- **日期**: 2026-04-09
- **状态**: ✅ 已接受
- **决策ID**: ADR-001

## 上下文 (Context)

Finda 是一款社交匹配类 App，对以下方面要求较高：
- 用户体验流畅性
- 推送通知可靠性
- 定位精度
- 相机/相册访问

需要决定技术栈选择。

## 考虑的方案

### 方案 A: React Native / Flutter (跨平台)
**优点:**
- 一套代码双端运行
- 开发效率高
- 热更新方便

**缺点:**
- 性能不如原生（列表滚动、动画）
- 第三方 SDK 集成麻烦（微信登录、支付宝、推送）
- 招聘难（需要懂跨平台的人才）
- 定制化 UI 成本高

### 方案 B: 原生开发 (Android Kotlin + iOS Swift)
**优点:**
- 性能最佳
- 生态系统成熟，SDK 丰富
- 招聘容易，人才储备充足
- 平台新特性第一时间支持

**缺点:**
- 需要维护两套代码
- 开发周期相对较长

## 决策

选择 **方案 B: 原生开发**

## 理由

1. **产品特性需求**: Finda 作为社交匹配 App，用户体验至关重要，原生性能优势明显
2. **团队现状**: 已有 Android 和 iOS 原生开发人员储备
3. **第三方集成**: 国内生态（微信登录、推送、支付）原生支持最完善
4. **长期维护**: 原生代码可维护性更好，便于后续迭代

## 后果

- ✅ 获得最佳性能和用户体验
- ✅ 第三方 SDK 集成简单
- ❌ 开发成本增加约 30%
- ❌ 需要同时维护两套代码

## 相关决策
- 后续评估是否需要共用 UI 组件库
- 考虑使用共享代码方案处理业务逻辑（如 Kotlin Multiplatform 或 C++）
