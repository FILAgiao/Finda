# Finda 撮合机制技术设计文档

> **版本**：V1.0
> **日期**：2026年4月6日
> **核心问题**：如何设计高效的约会撮合算法？

---

## 一、问题背景

### 1.1 核心矛盾

**无限的选择导致了零成交。**

传统社交软件（Tinder、Soul、探探）的问题：
- 用户左滑右滑看100个人，一个都约不到
- 信息过载导致决策瘫痪
- "赢家通吃"现象严重

### 1.2 设计目标

| 目标 | 说明 |
|------|------|
| 稀缺性 | 制造欲望 |
| 确定性 | 促成交易 |
| 高效率 | 降低系统复杂度 |

## 二、方案演进

### 方案一：3选1模式（交易所模式）

#### 2.1.1 时间维度：熔断机制

| 阶段 | 时长 | 说明 |
|------|------|------|
| 展示期 | 120分钟 | 卡片倒计时，结束自动销毁 |
| 响应期 | 30分钟 | 点击发起后，对方响应时限 |
| 过期重置 | - | 3个都没选，今日机会用尽 |

#### 2.1.2 机会维度：单线程约会

```
用户选择 A 并支付诚意金
    ↓
B、C 立即消失 → 释放回公共池
    ↓
全神贯注于 A 的沟通
    ↓
A 拒绝/超时 → 解锁 B、C 或补推新名额
```

#### 2.1.3 竞态条件问题

**问题描述**：
- A 看到 {B, C, D}，选择了 C
- B 看到 {A, E, F}，选择了 E
- 结果：A 和 B 互相错过

**这是分布式系统的经典 Race Condition 问题。**

---

### 方案二：镜像推送 + 批次处理

#### 2.2.1 镜像推送原则

> 如果系统判定 A 和 B 高度匹配，那么 A 的名单里一定有 B，B 的名单里也一定有 A。

**就像股票撮合**：买家A挂出价，卖家B挂要价，系统互相展示。

#### 2.2.2 批次化处理（The Batch/Pulse）

不是实时推送，而是像心脏跳动一样定时运行：

```
每隔 1 小时（或每早/中/晚）
    ↓
运行全局最优化算法（改进版 Gale-Shapley）
    ↓
Maximize Σ(Match_Score_i,j × Probability_i,j)
    ↓
打包分发匹配结果
```

#### 2.2.3 锁定机制

```
A 选择 C
    ↓
C 收到强提醒："A 已支付诚意金"
    ↓
C 的界面变化：A 的卡片闪烁/置顶
    ↓
C 在 30 分钟内不能选 X、Y
    ↓
必须先对 A 做 Yes/No 决策
```

**效果**：多对多 → 一对一精准决策

#### 2.2.4 分级漏斗算法

| 层级 | 名称 | 逻辑 |
|------|------|------|
| 第一层 | Geo-Time Grid | 只计算同城+48小时活跃用户 |
| 第二层 | Vector Similarity | Embedding向量化快速筛选 |
| 第三层 | Scarcity Distribution | 稀缺性动态分配 |

**稀缺性分配逻辑**：
- S级美女被1000人匹配 → 只推给3-5个最优质男生
- 同时，这位美女的名单里放上这几个男生
- 保证每个人看到的3个人都是"有机会约到的"

### 方案三：唯一推送模式（狙击手模式）⭐ 推荐

> 从"交易所模式"进化到"狙击手模式"

#### 2.3.1 核心理念

```
给用户选择 → 用户会犹豫
不给用户选择 → 用户会宿命感爆棚
```

#### 2.3.2 交互逻辑

```
12:00 PM（全城同步推送）
    ↓
所有人收到今日唯一的"搭子"名额
    ↓
60分钟决策窗
    ↓
双方决定是否质押诚意金
    ↓
结果：
  - 双向奔赴 → 成交
  - 单向/双向拒绝 → 今日机会作废
```

#### 2.3.3 波次追补逻辑

**问题**：推送一个没匹配上，效率太低？

**解决**：阶梯式释放

```
系统为每个人计算优先级队列 Q = {Match1, Match2, Match3}

第一波次 (12:00) → 推送 Match1
    ↓
失败（拒绝/超时）
    ↓
第二波次 (13:00) → 推送 Match2
    ↓
限制：同一时刻屏幕永远只有一个候选人
```

#### 2.3.4 时间匹配硬件化

```
用户点击"寻找搭子"前
    ↓
必须先勾选：[今天 18:00-20:00 我有空]
    ↓
AI 只在时间完全重合的人群里匹配
    ↓
消除沟通成本，见面是必然结果
```

## 三、完整撮合流程

### 3.1 Finda 交易撮合协议 V1.0

| 阶段 | 动作 | 说明 |
|------|------|------|
| 1. 挂单 | 用户输入意图 + 质押诚意金 | 15秒内完成 |
| 2. 暗池撮合 | AI后台匹配 | 时空+ELO+偏好 |
| 3. 推送 | 1个（或3个）候选人 | 模糊头像+信用等级 |
| 4. 锁定 | 30分钟独占决策 | 不能刷新/看别人 |
| 5. 成交 | 双方支付诚意金 | 高清照片+聊天解锁 |
| 6. 履约 | GPS到达验证 | 扫码/AI抽检 |
| 7. 清算 | 诚意金退还/赔付 | 信用分更新 |

## 四、营销与博弈论设计

### 4.1 "后悔药"机制（商业化点）

- 被抹除的 B、C 念念不忘？
- 消耗虚拟货币"捞回"
- 验证用户对特定特征的极度偏好

### 4.2 "拒绝"的体面化

| 传统显示 | Finda显示 |
|---------|----------|
| "对方拒绝了你" | "该搭子已被别人抢先预订" |

**效果**：保护用户自尊心，愿意继续尝试

### 4.3 确定性的降维打击

**口号**：不是"认识新朋友"，而是**"今晚一定有人陪你吃饭"**

### 4.4 唯一推送的优势

| 优势 | 说明 |
|------|------|
| 降低视奸风险 | 信息只被1人看到 |
| 消灭反荡妇机制 | 不再"货比三家" |
| 算法优化 | 完全二分图最大权匹配（KM算法） |
| 信誉溢价 | 爽约=明天匹配不到人 |

## 五、技术实现

### 5.1 核心算法

**1 vs 1 匹配 = 完全二分图的最大权匹配问题**

```
使用 KM 算法 (Kuhn-Munkres)

确保全城范围内，系统总满意度（成交额）最高
```

### 5.2 推荐模式

| 时间 | 模式 | 推送数量 |
|------|------|---------|
| 工作日 | 专业模式 | 1个 |
| 周末 | 狂欢模式 | 3个 |
