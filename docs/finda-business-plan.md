## 执行摘要

### 项目概述

**Finda（饭达/饭搭）** 是一款AI驱动的精准约会配对应用，核心理念是**"像股票交易所一样高效撮合约会需求"**。

与传统社交软件"左滑右滑、无限选择"的模式不同，Finda采用**"唯一推送+诚意金锁定"**的创新模式，每天只推送1个精准匹配对象，用博弈论机制确保约会履约率。

### 核心差异化

| 维度 | 传统社交软件 | Finda |
|------|-------------|-------|
| 交互模式 | 左滑右滑100人，约不到1个 | 一天一人，确定性决策 |
| 匹配效率 | 赢家通吃，普通人无机会 | 人人有饭吃，全局最优 |
| 隐私保护 | 广场公开，被视奸风险 | 暗池交易，双盲解锁 |
| 履约保障 | 放鸽子无惩罚 | 诚意金赔付+信用降权 |
| 目的匹配 | 猜测对方意图 | AI量化精准对齐 |

### 市场机会

- **千亿赛道**：中国社交APP市场规模135亿美金，约会细分市场年增长4.9%
- **竞品衰退**：Tinder/探探付费用户连续下滑，传统模式遇瓶颈
- **用户痛点**：76.8%单身青年渴望恋爱但追求"情价比"，拒绝无效筛选
- **技术窗口**：AI Agent时代到来，大模型可替代复杂表单

### 2.5 Finda差异化优势

|--------|----------|-----------|
| 推送模式 | 海量选择（看100人） | 唯一推送（1-3人） |
| 匹配算法 | ELO评分（赢家通吃） | KM算法（全局最优） |
| 隐私保护 | 公开展示照片 | 暗池交易（双盲解锁） |

- 暗池隐私：与竞品现有模式冲突，难以复制
- 信用体系：需长期数据积累，难以短期复制

## 三、产品与技术（整合技术架构+算法分析）

> **资料来源**：`/docs/technical-architecture.md`（工部产出）、`/docs/finda-matching-algorithm.md`、`/docs/finda-product-design.md`

### 3.1 产品核心设计

#### 核心理念

```
传统社交软件架构：
用户 → 左滑右滑 → ELO评分 → 推荐排序 → 用户选择 → 聊天 → 约会

Finda架构：
用户 → AI意图解析 → 批次撮合（KM算法） → 唯一推送 → 诚意金锁定 → GPS验证 → 信用清算
```

#### MVP切入点

**"约饭"是特洛伊木马**

| 优势 | 说明 |
|------|------|
| 高频刚需 | "吃"是最高频社交场景 |
| 合理推诿 | "我只是找人拼桌，不是相亲" |
| 时间确定性强 | 晚餐时间窗口固定（18:00-20:00） |
| 隐私保护天然 | 吃饭场景比约会更体面 |

### 3.2 技术架构概览

#### 系统架构

```
客户端层：iOS(SwiftUI) + Android(Kotlin) + Web(Next.js)
         ↓
API Gateway：Kong/AWS API Gateway（限流、认证、路由）
         ↓
服务层：用户服务 | 匹配服务 | 消息服务 | 支付服务 | AI意图解析 | 信用服务
         ↓
AI层：LLM意图识别 | 向量嵌入 | ELO评分引擎 | KM匹配算法 | 图像审核
         ↓
数据层：PostgreSQL主数据 | Redis缓存 | Elasticsearch向量检索 | Kafka事件流
```

#### 核心技术栈

| 层级 | 技术选型 |
|------|---------|
| 后端语言 | Go / Rust（高并发低延迟） |
| 数据库 | PostgreSQL + Redis + Elasticsearch |
| AI | Claude/GPT-4 API + OpenAI Embedding |
| 云服务 | AWS EKS + RDS + ElastiCache + MSK |
| 监控 | Datadog + CloudWatch + ClickHouse |

### 3.3 匹配算法核心

#### 匹配公式

```
Match_Score = α·Sim(Pref_A, Attr_B) + β·Sim(Pref_B, Attr_A) 
            + γ·TimeLoc_Score - δ·Friction_Cost
```

- α,β：双向偏好权重
- γ：时空匹配度
- δ：摩擦系数（信用惩罚）

#### 三级漏斗过滤

| 层级 | 名称 | 逻辑 |
|------|------|------|
| 第一层 | Geo-Time Grid | 只计算同城+时间窗口重叠用户 |
| 第二层 | Vector Similarity | Embedding向量化快速筛选Top 50 |
| 第三层 | Scarcity Distribution | 稀缺性动态分配，S级用户限曝光 |

#### KM算法

**完全二分图最大权匹配**

- 确保全城范围内，系统总满意度（成交额）最高
- 避免"赢家通吃"，让普通人也有机会
- 匹配结果全局最优而非局部最优

### 3.4 诚意金机制

#### 流程设计

```
用户质押诚意金（50-200元）
    ↓
匹配成功 → 双方锁定诚意金
    ↓
见面打卡 → GPS验证/AI抽检
    ↓
结果：
  - 成功见面 → 双方诚意金退还 + 信用分+2
  - 一方爽约 → 爽约方诚意金赔付给对方 + 信用分-30
```

#### 博弈论效果

- 过滤90%口嗨党和骗子
- 制造承诺感，降低爽约率
- 建立信用体系，形成正向循环

### 3.5 暗池交易模式

#### 设计理念

> 像股票暗池交易一样，信息不公开，撮合后才解锁

#### 流程

```
高隐私用户（公务员、教师）
    ↓
照片和资料不在广场公开展示
    ↓
AI后台匹配，判定高度契合
    ↓
双方确认"挂单" → 双盲解锁信息
    ↓
见面决策
```

#### 效果

- 吸引高质量、高隐私要求用户
- 降低视奸风险
- 形成差异化竞争壁垒

### 5.1 品牌定位

**对外口号**：
> "一天只推一个人，月老点名就是他"

**品牌调性**：
- 正向、阳光、健康（表面）
- 精准、高效、不废话（内核）
- 体面、私密、可信赖（气质）

#### 第一优先级：高隐私需求群体

- 公务员、教师、企业高管、金融从业者

- 互联网大厂员工、医生护士、创业者

- 抑郁症患者、短期陪伴需求者、特定兴趣爱好者
- 市场规模：一线城市约300万人

| 定性 | 法律后果 | 合规要求 |

| 押金 | 合法，需按约定退还 | 需明确保管方式 |
| 资金池风险 | 需第三方存管 | 建议与支付平台合作 |

1. 明确定金性质：在用户协议中明确诚意金为"履约保证金"
