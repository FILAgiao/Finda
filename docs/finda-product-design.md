## 一、社会背景与问题分析

### 1.1 市场痛点

目前约会是一种**低效率配对模式**。与商品交易类比：

- 街上偶遇、家人朋友介绍对象 = 路边摆摊消费
- 人的口味挑剔、需求即时
- 地摊很难买到迫切想要的东西

**约会市场是低效的交易市场**：买卖双方无法快速撮合，导致：

| 异常现象 | 根因 |
|---------|------|
| 超出正常市场价格的高端卖淫 | 有效供给不足 |
| 天价彩礼 | 匹配效率低，供需失衡 |

**核心假设**：如果能像股票交易单一样量化约会需求（时间、颜值类型、职业、学历、目的等），不断撮合交易，约会市场会越来越有效。

## 二、现有社交软件的问题

### 2.1 赢家通吃

- 特别帅的帅哥/美女垄断市场
- 普通人很难约到对象
- 理想市场：人人有饭吃，人人有对象

### 2.2 目的性难以匹配

| 短期目的 | 长期目的 |
|---------|---------|
| 约炮为主，陪伴为辅 | 长期恋爱、炮友、线上陪伴综合体 |

**问题**：
- 不知道对方真实需求
- 通过聊天、建立安全感、左滑右滑等复杂前置环节 → 用户流失
- 小众需求难以匹配（如只想聊天、抑郁症患者）

### 2.3 信息交互问题

**个人信息暴露**：
- 大广场式公开展示
- 有社会身份的人（公务员、教师等）不愿参与
- 怕被视奸，流失大量客户

**反荡妇/骚男机制**：
- 必须假装正经、矜持
- 过早暴露真实需求 → 触发防御机制
- 形成繁文缛节，打击积极性

### 2.4 信息准确性问题

- 骗子多
- 缺乏验证机制

### 2.5 道德性问题

- 约炮软件被认为不道德
- 需要营造正向、积极形象
- 否则做再大也只是"妓院/鸭馆"

## 三、产品核心设计

### 3.1 核心目的

**增强约会市场的有效性**

### 3.2 核心原理

股票交易单是一维数据（出价高低），约会需求是多维量化数据。

### 3.3 数据维度

#### 🔴 重要（食色）

| 维度 | 说明 |
|------|------|
| 约会目的 | 网聊/线下陪伴/速配约炮 |
| 对方性别、年龄、颜值类型 | 照片选择，大类划分 |
| 想见面地点 | 可选，避免过于复杂 |
| 时间空闲 | 核心匹配维度 |

#### 🟡 一般（脑）

爱好、职业、学历、性格

#### 🟢 待定（魂）

世界观

#### 变与不变

- **容易变**：空闲时间、约会目的、地点（激励用户主动更新）
- **不容易变**：性别、年龄

### 3.4 用户交互模式

1. **输入框交互**：自然语言描述需求
2. **卡片交互**：左滑右滑记录颜值偏好
3. **颜值检测**：活体检测 + 卡颜

### 3.5 首页推送模式

- 推送"3天内有空吗？""最近想约吗？"
- 展示"今天为你寻找了多少配对机会"
- 允许用户微调算法（今天想～只要女的/男的就行）
- 少量精准推送，有查看限制
- 放鸽子降低信用分

### 3.6 核心功能模块

| 模块 | 功能 |
|------|------|
| 首页 | 推送配对机会 |
| 我的行程 | 本周聚会安排 |
| 个人主页 | 展示照片、评价 |
| 精选（待定） | 附近人评价/帖子 |

## 四、关键问题讨论

### Q1：是否锚定双方价值？（普男普女 vs 帅哥美女）

**结论**：80/20法则

| 比例 | 策略 |
|------|------|
| 80% | 严格锚定ELO价值（颜值+综合评分），确保成功率高 |
| 20% | 随机蜜糖（Variable Reward），推送略高于评分的对象，制造惊喜感 |

**增长黑客**：早期冷启动可引入托儿，度过冷启动后迅速切换纯真实用户。

### Q2：MVP切入点选择

**结论**：死磕"约饭"

- "吃"是最高频、最刚需、最自然的社交场景
- "饭搭"是完美的**特洛伊木马**
- 提供"合理推诿"属性："我只是找人拼桌，不是在相亲"
- 切口缩到极小："今晚7点，XX商圈，我想吃火锅，谁来？"

### Q3：如何识别Sugar Daddy等复杂需求？

**结论**：抛弃表单，拥抱大模型Agent

- 让用户用自然语言写需求
- 后端AI Agent自动提取：{Budget, Intent, Power_Dynamic}
- 表面维持体面，后台精准匹配

### Q4：是否开启约会评价功能？

**结论**：不做"淘宝式公开评价"，采用"隐性信用评级"

**问题**：
- 公开评价 = 隐私泄露、网暴、恶意报复
- 降低用户约会意愿

**方案**：双盲私密问卷
- 照片真实度（1-5星）
- 行为举止是否有礼貌
- 是否想再见

数据吸收进隐性ELO评分和信用体系，连续差评 → 降权/Shadowban。

## 五、算法与博弈论设计

### 5.1 动态价值模型（ELO Rating）

```
Match(A,B) = α·Sim(Pref_A, Attr_B) + β·Sim(Pref_B, Attr_A) + γ·f(Time,Loc) - δ·Friction
```

- α,β：双向偏好权重
- γ：时空匹配度
- δ：摩擦系数/信用分惩罚

### 5.2 防鸽机制（Anti-Flaking）

引入博弈论"承诺机制"：

| 场景 | 处理 |
|------|------|
| 顺利见面打卡 | 诚意金原路退回，或转化为高级算力/曝光度 |
| 一方爽约 | 诚意金赔付给被放鸽子的一方 |

**效果**：过滤90%口嗨党和骗子

## 六、隐私保护（暗池交易）

针对公务员、教师等高隐私需求用户：

- 照片和资料不在广场公开展示
- AI判定高度契合 + 双方确认"挂单" → 双盲解锁
- 吸引高质量、高隐私要求用户

## 七、下一步行动计划

### MVP冲刺

1. **极简交互**：首页只有一个大按钮"今天我想找个搭子"
2. **强制限制**：每天只推3个人，制造稀缺感
3. **核心开发**：
   - 大模型意图解析
   - 基于地理位置/时间的匹配算法
   - 前端UI最简即可

### 合规红线

- 绝对砍掉"SM调教、体位"等显性标签
- 表面阳光、健康、社交
- 底层"食色"需求由AI隐性处理
