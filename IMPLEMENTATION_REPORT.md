## 2. 项目背景

### 2.1 Finda 产品定位
**Finda 是一款 AI 驱动的约会/饭搭撮合平台**，解决传统约会软件的低效问题：

**传统约会软件的问题**:
- 左滑右滑 → 选择困难，永远觉得"下一个更好"
- 聊天预热 → 聊一周不见面，浪费时间
- 被鸽率高 → 对方不认真，临时爽约
- 隐私暴露 → 大广场晒照片，公务员/教师不敢用

**Finda 的解决方案**:
- **一天只推一个人** → 没得选，只能认真考虑
- **即时撮合** → 基于"今晚想吃火锅"实时匹配
- **诚意金质押** → 爽约方赔付，确保认真
- **暗池模式** → 信息仅匹配双方可见

### 2.2 原有状况
- 已有 Android/iOS 原生代码库
- 已有基础技术文档（架构、算法、竞品分析）
- 缺少系统化的需求管理和用户洞察
- **需要明确**: 产品定位是约会撮合，不是服务撮合

### 2.3 目标
建立一套完整的项目管理体系：
1. 结构化需求管理（Backlog）
2. 技术决策记录（ADR）
3. 约会场景用户心理洞察（Psychology Map）
4. 增长实验设计（Experiments）
5. AI 协作规则（.cursorrules）

#### 3.2.2 用户旅程情绪地图（约会场景）

**主动方关键情绪点**:
1. **发布需求** → 期待
2. **等待匹配** → 焦虑 ⚠️ (需优化)
3. **收到确认** → 兴奋 ★ (峰值点)
4. **见面约会** → 紧张→愉悦
5. **完成评价** → 满意

**响应方关键情绪点**:
1. **收到推送** → 惊喜 ★ ("有人想请我吃饭")
2. **查看资料** → 评估
3. **决策** → 选择困难 ⚠️ (需简化)
4. **接受后** → 期待
5. **见面后** → 评价

**关键优化策略**:
- 等待焦虑 → AI月老进度可视化
- 选择困难 → 每天只推1个人
- 被拒失落 → "缘分未到，明天再来"

## 4. 关键创新点

### 4.1 约会撮合 vs 服务撮合的本质区别

| 维度 | 服务撮合 | 约会撮合 |
|------|---------|---------|
| 交易性质 | 单向（我买你服务） | 双向（互相选择） |
| 核心痛点 | 质量不确定 | 被鸽、选择困难、隐私暴露 |
| 解决方案 | 评价系统 | 诚意金、每天1人、暗池模式 |
| 成功标准 | 服务完成 | 双方满意、可能二次约会 |

### 4.2 核心差异化功能

1. **每天只推一个人**
   - 解决选择困难
   - 创造稀缺性和专属感
   - 迫使用户认真考虑

2. **诚意金质押**
   - 解决"被鸽"问题
   - 爽约方赔付，筛选认真用户
   - 建立信任基础

3. **暗池模式**
   - 保护隐私，区别于Tinder
   - 吸引公务员/教师等精英用户
   - 降低社交压力

4. **AI月老撮合**
   - 基于意图匹配，而非外貌滑动
   - "今晚想吃火锅"→匹配合适对象
   - 降低反荡妇/反骚男机制

## 5. 专家深度分析（Critical Review）

### 5.1 技术架构师视角 🔧

#### 问题1: 实时撮合的技术债务

**当前设计缺陷**:
- 每小时运行一次全局撮合 → 延迟太高（用户需等待1小时）
- Redis锁 + 全局匹配 → 单机瓶颈，无法水平扩展
- 地理位置分区 → 同城但跨区的人无法匹配

**专家解决方案**:
```
┌─────────────────────────────────────────┐
│ 改为"流式撮合"架构                      │
│                                         │
│ 用户发布需求 ──▶ Kafka ──▶ Flink实时计算 │
│                              ↓          │
│ 实时匹配候选池 ◀── 内存网格 ◀──┘          │
└─────────────────────────────────────────┘

优化点:
- 延迟从小时级降到秒级
- 使用Redis Cluster + Redission分布式锁
- 引入GeoHash算法，10km内精确匹配
```

**MVP建议**: Phase 1可用简化版（5分钟批次），Phase 2引入流式架构

---

#### 问题2: 诚意金的支付合规风险

**风险识别**:
- 用户资金托管需要**支付牌照**
- 爽约赔付涉及"资金结算"，涉嫌**二清违规**
- 没有牌照最多只能做"虚拟积分"

**解决方案对比**:

| 方案 | 描述 | 优点 | 缺点 |
|------|------|------|------|
| **A. 持牌机构合作** | 引入银行/第三方支付托管 | 完全合规 | 成本高，需分润15-20% |
| **B. 虚拟积分模式** | 用户充值积分，用积分支付 | 无牌照风险 | 用户感知弱，效果打折 |
| **C. 餐厅预付模式** | 预付到餐厅，不经过平台 | 餐饮SaaS模式，规避资金池 | 依赖餐厅合作 |

**专家建议**: MVP阶段采用**方案B（虚拟积分）**，验证模式后转向**方案C（餐厅预付）**

### 5.2 产品经理视角 📱

#### 问题1: 双边市场的冷启动死亡陷阱

**冷启动公式**:
```
有效匹配 = 男性用户数 × 女性用户数 × 匹配成功率
```

**死亡螺旋**:
```
初始用户少 → 匹配成功率低 → 用户流失 → 用户更少
     ↑___________________________________________↓
```

**专家解决方案**:

```
┌────────────────────────────────────────┐
│ 冷启动三阶段策略                        │
├────────────────────────────────────────┤
│ Phase 1 (0-1000用户): 邀请制+人工撮合   │
│ - 邀请码机制，筛选种子用户              │
│ - 运营人员人工撮合（假装AI）            │
│ - 引入"托儿"（付费美女用户）            │
│ - 男女比例严格控制 4:6                  │
├────────────────────────────────────────┤
│ Phase 2 (1000-10000): 城市深耕          │
│ - 单城市打透（北京/上海）               │
│ - 线下活动导流（单身饭局）              │
│ - KOL合作（小红书情感博主）             │
├────────────────────────────────────────┤
│ Phase 3 (10000+): 算法驱动增长          │
│ - 口碑裂变（邀请奖励）                  │
│ - 内容营销（成功故事）                  │
└────────────────────────────────────────┘
```

**MVP关键决策**: 前1000用户**仅限女性邀请**，男性需审核（学历/职业认证）

---

#### 问题2: 女性用户体验的致命缺陷

**当前设计问题**:
- "每天推1人" → 女性选择权被剥夺
- "诚意金" → 男性付费，女性免费 → 被物化感
- "暗池" → 看不到照片 → 安全感缺失

**女性核心需求（优先级排序）**:
1. 安全感（不是变态/渣男）
2. 选择权（我想选谁，不是被分配）
3. 被尊重（不是被"约"的对象）

**专家解决方案**:

```
┌─────────────────────────────────────────┐
│ 双模式设计（用户自选）                   │
├─────────────────────────────────────────┤
│ 模式A: "被选择"模式（当前设计）          │
│ - 适合：被动型用户                      │
│ - AI推送，等待确认                      │
│ - 女性用户默认关闭                      │
├─────────────────────────────────────────┤
│ 模式B: "主动选择"模式                    │
│ - 适合：主动型用户                      │
│ - 女性每天可以浏览3个候选人             │
│ - 男性保持"每天被分配1人"               │
└─────────────────────────────────────────┘

核心原则：女性永远有选择权，不是被分配的客体
```

```
│ - 双方确认"照片真实"                    │
│ - 不通过则重新匹配                      │

│ Step 2: 语音破冰（约会前）               │
│ - AI生成话题推荐                        │
│ - 双方语音预热                          │
│ - 建立基本默契                          │
│                                         │
│ Step 3: 线下见面                        │
│ - 降低"开盲盒"风险                      │
└─────────────────────────────────────────┘
```

**游戏化留存设计**:

### 5.4 投资人/商业分析师视角 💰

#### 问题: 商业模式的脆弱性

**当前模式分析**:
```
收入 = 用户数 × 爽约率10% × 50元 × 抽成20%
     = 1元/用户

需要100万日活才能月入300万
```

**致命问题**: 用户越多，爽约率越低（筛选效应），收入反而下降

**专家解决方案**:

```
┌─────────────────────────────────────────┐
│ 多元收入模型                            │
├─────────────────────────────────────────┤
│ 1. 会员订阅（主要收入）60%               │
│    - VIP：99元/月，免诚意金+优先匹配      │
│    - 预计付费率5%，ARPPU 99元            │
│                                         │
│ 2. 增值服务（高频收入）25%               │
│    - 超级曝光：让更多人看到你（20元/次）  │
│    - 精准筛选：按学历/收入筛选（10元/次） │
│    - 礼物商城：虚拟礼物（抽成30%）        │
│                                         │
│ 3. B端收入（稳定收入）15%                │
│    - 餐厅导流：成交抽成5%                │
│    - 活动运营：线下单身活动（门票分成）    │
└─────────────────────────────────────────┘
```

## 7. 专家级关键决策建议

### 决策1: 先做"饭搭"还是"约会"？

**专家建议**: 明确做"饭搭"，淡化"约会"

| 维度 | 饭搭 | 约会 |
|------|------|------|
| 用户压力 | 低（只是吃饭） | 高（找对象） |
| 成功率 | 高（吃饭容易成功） | 低（找对象是长期过程） |
| 口碑传播 | 容易（有趣的经历） | 难（隐私敏感） |
| 监管风险 | 低 | 高（容易被认定为约炮） |

**Slogan调整**:
- ❌ "一天只推一个人，月老点名就是他"（太正式）
- ✅ "今天想吃火锅，AI帮你找个饭搭子"（轻松）

---

### 决策2: 先做男性还是女性？

**专家建议**: 先做女性，控制男性涌入

**原因**:
- 约会软件永远缺女性
- 有女性自然有男性
- 女性体验好才能留住

**执行**:
- 前1000用户仅限女性邀请
- 男性需审核（学历/职业认证）
- 保持男女比4:6

---

### 决策3: 先做哪个城市？

**专家建议**: 北京（或上海）

**原因**:
- 公务员/教师密度高
- 单身率高
- 餐饮丰富
- 媒体/PR资源丰富

**策略**:
- 单城市打透（不做全国）
- 与头部餐厅合作
- 地推+小红书种草

## 8. 修正后的MVP排期（务实版）

### 原MVP（过于理想化）vs 专家MVP（务实可执行）

| 阶段 | 原MVP | 专家MVP | 调整原因 |
|------|-------|---------|---------|
| Week 1-2 | 撮合算法V1 | **人工撮合** | 冷启动期算法无数据，不如人工精准 |
| Week 3-4 | 诚意金支付 | **虚拟积分系统** | 规避支付牌照风险 |
| Week 5-6 | 暗池模式 | **简化隐私模式** | 先做基础隐私，逐步增强 |
| Week 7-8 | 开放注册 | **邀请制1000人** | 控制质量，保证体验 |

### 详细排期

```
Week 1: 基础搭建
├── 用户注册/登录
├── 基础资料填写
└── 人工撮合后台（运营人员操作）

Week 2: 撮合流程
├── 需求发布（吃什么/何时/哪里）
├── 人工匹配（运营手动匹配）
├── 双方确认
└── 联系方式交换

Week 3: 虚拟积分
├── 积分充值（微信支付）
├── 积分消费（发布需求消耗）
└── 积分退还（成功见面退还）

Week 4: 评价体系
├── 约会后评价
├── 信用分计算
└── 人工审核评价

Week 5-6: 算法MVP
├── 简单匹配算法（地理位置+时间）
├── 每日1人推送
└── A/B测试（人工vs算法）

Week 7-8: 冷启动
├── 邀请码系统
├── 种子用户招募（女性优先）
├── 运营人员培训
└── 数据收集与迭代
```

### 冷启动策略（并行执行）

```
Phase 1 (Week 1-4): 种子用户积累
├── 目标：100个女性种子用户
├── 渠道：小红书KOL、朋友推荐、线下活动
├── 运营：1v1服务，人工撮合
└── 激励：免费VIP体验

Phase 2 (Week 5-8): 控制增长
├── 目标：1000用户（600女/400男）
├── 男性准入：需邀请码+学历/职业认证
├── 质量控制：每周新增不超过200人
└── 托儿引入：付费美女用户（10-20人）

Phase 3 (Week 9+): 算法接管
├── 算法匹配率 > 人工匹配率
├── 逐步放开注册
└── 开启口碑裂变
```
